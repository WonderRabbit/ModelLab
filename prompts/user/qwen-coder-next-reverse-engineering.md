# Qwen-Coder-Next 역공학 분석 프롬프트 (React/Vertex + Spring + SAP)

목적: 기존 코드/로그/API 스펙을 바탕으로 React(또는 React+Vertex AI 연동), Spring 백엔드, SAP 인터페이스를 포함한 E2E 비즈니스 플로우를 역공학하여 아래 산출물을 생성한다.
- 시퀀스 다이어그램
- UI 설계 분석
- 데이터 플로우 다이어그램(DFD)
- 유저 스토리

권장 사용 방식:
1. 이 문서를 사용자 프롬프트 템플릿으로 복사한다.
2. `<PROJECT_CONTEXT>`와 `<EVIDENCE_INPUT>`를 실제 자료로 치환한다.
3. 1차 결과 후 `검증 질문`을 코드/도메인 담당자에게 확인하고 2차 보정한다.

---

## 프롬프트 템플릿

```md
역할:
너는 시니어 소프트웨어 아키텍트이자 역공학 분석가다.
목표는 기존 시스템(React/Vertex, Spring, SAP)의 실제 동작을 증거 기반으로 구조화하는 것이다.

작업 범위:
- 프론트엔드: React 화면, 상태관리, API 호출, 사용자 인터랙션
- AI/오케스트레이션: Vertex AI 호출 경로(있을 경우), 프롬프트/응답 처리
- 백엔드: Spring Controller-Service-Repository 및 외부 연동
- 엔터프라이즈 연동: SAP RFC/BAPI/IDoc/OData/파일 인터페이스
- 횡단 관심사: 인증/인가, 에러처리, 로깅, 재시도, 트랜잭션, 배치

입력:
<PROJECT_CONTEXT>
- 시스템 개요:
- 주요 도메인:
- 분석 대상 기능:
- 기술 스택/버전:

<EVIDENCE_INPUT>
- 코드 경로/스니펫:
- API 명세(REST/gRPC/OpenAPI):
- DB 스키마/ERD:
- SAP 인터페이스 문서:
- 로그/트레이스/모니터링:

엄격 규칙:
1) 관찰된 사실과 해석을 분리하라.
2) 근거 없는 단정 금지. 모든 핵심 주장에 근거 ID를 연결하라.
3) 불확실성 라벨을 반드시 사용하라: evidence / inference / hypothesis.
4) 누락 정보는 질문으로 명시하고, 임의 보완 시 가정을 기록하라.
5) 출력은 반드시 아래 섹션 순서를 지켜라. 추가 섹션 금지.

출력 형식(고정):

# 1. 분석 스코프 및 가정
- 분석 대상:
- 제외 범위:
- 주요 가정:

# 2. 증거 인벤토리
| Evidence ID | 출처 | 요약 | 신뢰도(evidence/inference/hypothesis) |
|---|---|---|---|

# 3. As-Is 비즈니스 로직 요약
- 핵심 유스케이스 3~7개를 요약하라.
- 각 유스케이스마다: 트리거, 입력, 핵심 처리, 결과, 예외 흐름.

# 4. 시퀀스 다이어그램 (Mermaid)
- 최소 2개:
  - 정상 흐름(Happy Path)
  - 오류/보상 흐름(Exception/Compensation)
- 참여자에 React UI, Spring API, SAP, (해당 시) Vertex AI 포함.

# 5. UI 설계 분석
- 화면/컴포넌트 트리(페이지 > 섹션 > 컴포넌트)
- 상태 전이(초기/로딩/성공/오류/재시도)
- 사용자 행동과 백엔드 호출 매핑
- 접근성/성능/오류 메시지 관점 리스크

# 6. 데이터 플로우 다이어그램 (DFD)
- Context Level(레벨 0)
- Level 1(주요 프로세스 분해)
- 데이터 저장소(DB/SAP/캐시/파일)와 데이터 객체 명시
- 민감정보(PII/재무정보) 태깅

# 7. 유저 스토리 및 인수기준
- 최소 5개 스토리, 형식:
  - As a <role>, I want <goal>, so that <value>
- 각 스토리별 인수기준(Given-When-Then) 2개 이상
- 백오피스/SAP 운영자 관점 스토리 1개 이상 포함

# 8. 갭/리스크/기술부채
- 역공학 중 확인된 불일치(문서 vs 코드)
- 장애 유발 가능 지점 Top 5
- 단기/중기 개선안

# 9. 검증 질문 및 다음 단계
- 확인이 필요한 질문 5~10개
- 추가로 필요한 증거 목록
- 다음 분석 반복 계획(1주 이내 실행 가능한 단위)

품질 체크리스트:
- 다이어그램은 문법적으로 유효한 Mermaid여야 한다.
- 모든 다이어그램 요소가 텍스트 설명과 일치해야 한다.
- SAP 연계는 인터페이스 유형과 실패 시 재처리 전략을 포함해야 한다.
- Vertex AI가 없는 시스템이면 "해당 없음"으로 명시하고 대체 경로를 설명하라.
```

## 활용 팁
- 코드 근거를 줄 때 `파일경로:라인` 형식을 강제하면 추적성이 올라간다.
- 1차 산출물에서 `hypothesis` 비율이 높으면, 로그/트레이스부터 추가 수집하는 것이 효율적이다.
- 유저 스토리는 구현 태스크가 아니라 사용자 가치 중심 문장으로 유지한다.
